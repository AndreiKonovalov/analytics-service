# Demo план: N+1 проблема и способы решения

## Цель
Показать эффект N+1 запросов, а затем сравнить с оптимизированными вариантами: EntityGraph и map‑проекцией.

## Подготовка
- Запустите приложение и убедитесь, что в базе есть клиенты/счета/транзакции.
- Включите логирование SQL (если нужно) в `application.yml` для наглядности.

## Шаги демонстрации

### 1) Сначала показать проблему (N+1)
**Endpoint:** `GET /api/v1/analytics/clients/naive`

**Что ожидать:**
- В логах видно большое количество запросов (1 + N + N*M ...).
- Время выполнения заметно выше.

### 2) Затем показать оптимизацию через EntityGraph
**Endpoint:** `GET /api/v1/analytics/clients/optimized`

**Что ожидать:**
- В логах меньше запросов за счет жадной загрузки связей через `EntityGraph`.
- Время выполнения ниже.

### 3) Показать вариант с проекцией
**Endpoint:** `GET /api/v1/analytics/clients/summary`

**Что ожидать:**
- Запрос возвращает агрегированные данные без полной загрузки сущностей.
- Минимальный объем данных и запросов за счет map‑проекции на уровне JPQL.

## Итоговое сравнение
- **Naive** демонстрирует N+1.
- **Optimized (EntityGraph)** уменьшает количество запросов при загрузке детализации.
- **Summary (Projection)** возвращает только нужные поля и агрегаты, минимизируя нагрузку.
